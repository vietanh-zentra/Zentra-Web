Đúng, chỗ “bắt buộc Neon” trong bản trước là quá cứng nếu bạn đã có source code của web khách hàng. Tài liệu gốc ghi Dũng phải “Setup Neon”, nhưng đồng thời cũng ghi rõ Hoà phải khảo sát hạ tầng AWS hiện tại của khách trước khi deploy, xem họ đang dùng gì rồi mới ghép và triển khai. Vì vậy, với thông tin mới bạn vừa nói, cách đúng hơn là: database phải bám theo kiến trúc có sẵn của source code khách hàng, không mặc định ép sang Neon; Neon chỉ nên là phương án tạm nếu trong codebase chưa có DB dùng được.

Tôi làm lại chi tiết theo hướng đó cho bạn.

III. Nguyên tắc phối hợp để không lệch code

Tài liệu gốc đã nhấn mạnh các điểm:

Hoà và Dũng làm độc lập nhưng phải thống nhất format dữ liệu từ đầu
Dũng phải push code hằng ngày
Hoà phải review liên tục, tích hợp dần dần, không đợi đến cuối.

Tôi chuyển 3 điểm đó thành bộ quy tắc vận hành rõ ràng hơn:

Rule 1: Contract trước, code sau

Trước khi ai code sâu, phải chốt:

payload account_info
payload trades
payload open_positions
payload summary
error response format
kiểu dữ liệu cho từng field
field bắt buộc / nullable
timezone = UTC.
Rule 2: Source code khách là nguồn sự thật

Dũng phải đọc source code hiện có rồi kết luận:

DB hiện tại là gì
bảng nào đã có sẵn
service nào có thể tái sử dụng
endpoint nào có thể mở rộng thay vì viết mới

Không được ép DB mới chỉ vì tài liệu cũ đề xuất Neon.

Rule 3: Hoà cung cấp payload thật càng sớm càng tốt

Dũng không nên build backend dựa trên JSON ví dụ duy nhất trong spec.
Hoà phải tạo payload mẫu thật từ MT5 sớm để:

test insert
test parsing
test pagination/filter
phát hiện field mismatch.
Rule 4: Mỗi ngày đều có điểm ghép

Không làm kiểu “chia đôi rồi cuối tuần mới ghép”.
Mỗi ngày phải có ít nhất một bước:

Dũng đẩy API/schema mới
Hoà test bằng payload thật
phát hiện lệch và sửa ngay.
